## 0. Context / thesis

Direction A from the survey is authoritative: migrate now to self-hosted NATS, drop the relay transport/infra layer, and keep `@hangar-bridge/shared` + `@hangar-bridge/peer-agent` as the runtime substrate. This is consistent with `docs/VISION.md` ¬ß5, where NATS is already the acknowledged likely anchor and Direction (2) is a bounded-curiosity detour, not an ongoing open question.

The thesis is explicit: after cutover, system authority sits in the envelope schema + peer-agent + subject/transport conventions, with the arbitration protocol deferred to a follow-up plan. The six wire kinds in `packages/shared/src/envelope.ts` remain the format contract.

Cross-validation correction from the survey is non-negotiable. The relay‚Äôs retirement boundary is smaller than intuition: NATS does **not** buy permanent idempotency, correlation integrity, durable correlation memory, or arbitration protocol correctness by itself. The plan treats these as peer-agent/substrate work, plus a future protocol plan.

The `packages/shared/src/ulid.ts` monotonic factory and the relay resume cursor path are retained for now in current shared protocol behavior, but they are **OBSOLETE in the retired relay path** where SSE reconnect uses `?since=<ulid>` today. JetStream sequence numbers and KV revision checkpoints replace that cursor in the future NATS path.

## 1. Problem

`docs/architecture.md` and `SUBJECT_ROUTING_SPEC.md` define a one-relay physical topology with subject ACL chokepoints, protocol invariants, and per-kind semantics. Migration must preserve all protocol guarantees while removing relay dependence and replacing it with a single NATS anchor. The primary risk is loss of guarantees that were previously provided by relay code paths, especially sender identity, durable dedupe, presence truthfulness, and kind-specific backfill.

The immediate objective is to retire relay transport/infra now, not later, without weakening security or semantics established by `CLAUDE.md` and `docs/architecture.md`.

## 2. OKR / KRs

- **Objective:** complete migration to NATS-backed transport + JetStream + KV while keeping envelope protocol, six-kind wire semantics, and security invariants intact; keep relay source as transitional compatibility path until final phase.
- **KR1 (Direction A closure):** all transport-level functionality in `@hangar-bridge/relay` is retired by final phase and replaced by operationally equivalent NATS paths with testable gates.
- **AC1 ‚Äî JetStream single-node and durable fsync:** `nats-server` deployed as single-node R1 streams with `replicas: 1` and `sync_interval: always` in the shipped config.
- **AC2 ‚Äî Static NKey auth and deny-by-default subject publish ACL:** every fleet user authenticates by `nkey` only (no password/token fallback), publish allow is EXACTLY `["fleet.<handle>.>"]`, deny-by-default (per the ¬ß2.6 subject scheme; the sender token is NATS-enforced true identity). This transport-level subject-prefix ACL bounds WHERE a peer may publish; it is DISTINCT from the app-side envelope-subject namespace ACL (AC11) ‚Äî NATS cannot inspect `Envelope.subject`. Request-reply reply subjects live inside the responder's own `fleet.<handle>.>` prefix (¬ß2.6), so no `_INBOX.>` grant is needed for envelopes. Gate: a named config test asserts each user has an `nkey`, no username/password or `token` credential entries exist (nkey-only), and the ENVELOPE `publish.allow` entry is exactly `fleet.<handle>.>` with deny-by-default. The additional minimal JetStream/KV control grants (`$JS.API.>`/`$JS.ACK.>`, the KV-bucket subjects, own `_INBOX.<nuid>.>`) are a separate tested allow-list (¬ß2.6) ‚Äî the test asserts they are present but NOT broadened to `fleet.>` or `>`.
- **AC12 ‚Äî Subscribe chokepoint preserved (only the addressed recipient receives).** Each user's ENVELOPE `subscribe.allow` lane is exactly `["fleet.*.to.<handle>.>", "fleet.*.to.team.>"]`, deny-by-default (¬ß2.6); the ONLY additional grant is the user's own scoped `_INBOX.<nuid>.>` control-reply subject (never a bare `_INBOX.>`). This reproduces the relay subscribe chokepoint (architecture ¬ß5.5 / SUBJECT_ROUTING_SPEC ¬ß4.3): a peer cannot subscribe to another peer's `to.<other>` traffic. Gate: (a) `nats-config.test.ts` asserts each user's envelope `subscribe.allow` matches the recipient-scoped pattern and nothing broader (no `fleet.>`, no bare `_INBOX.>`), and that any `_INBOX` grant is the scoped own-`_INBOX.<nuid>.>` form only; (b) a negative integration test proves a peer subscribing to `fleet.*.to.<other-handle>.>` is DENIED / receives nothing; (c) a self-delivery test proves a broadcaster does NOT receive its own `to.team` message (¬ß2.6 self-drop); (d) `nats-config.test.ts` asserts the FLEET roster handles (`fleet-roster.json`) and the conf FLEET NKey users are the EXACT same set (no orphan fleet user, no roster-only handle) ‚Äî the `$SYS`/admin credential is excluded from both the roster and this equality check; (e) a test asserts no fleet handle equals the reserved broadcast token `team` (¬ß2.6).
- **AC13 ‚Äî Wire-subject grammar authority (`parseFleetSubject`).** The peer-agent validates every inbound wire subject as exactly `fleet.<sender>.to.<recipient>.<kind>` (¬ß2.6). Split by enforcement layer: the `<sender>` = authenticated-principal guarantee is a **NATS ACL integration test** (a peer physically cannot publish under another handle because `publish.allow=fleet.<handle>.>`), NOT a peer-agent check (the peer-agent can't observe the principal and must not read `Nats-Request-Info`, AC10). The **peer-agent** gate: negative tests reject (a) malformed / extra-token subjects, (b) `<recipient>` ‚âÝ local handle/`team`, (c) `<kind>` ‚âÝ `Envelope.kind`, (d) `<recipient> == team && <kind> ‚àâ {chat, presence_update}` (hard reject ‚Äî so `task_dispatch`/`task_result`/`permission_*` to `@team` are all rejected, not just `task_dispatch`). The peer-agent DERIVES `from` from the `<sender>` token; identity/routing come from the wire subject, never the forgeable envelope body.
- **AC11 ‚Äî App-side envelope-subject ACL preserved (SUBJECT_ROUTING_SPEC.md fail-closed namespace ownership).** The fork's subject-ACL (publisher-owns AND recipient-owns the namespace; `subject != null ‚áí concrete `to`, never `@team`; subjected reactive/system kinds rejected; ack-channel `subject=null`) CANNOT be enforced by NATS subject permissions (which see only the wire subject, not the JSON `Envelope.subject`). It MUST relocate into the peer-agent (app-side), with the sender identity derived from the authenticated `fleet.<handle>.>` publish subject (AC2b), never from envelope `from`. The **ownership authority** (which handle owns which namespace) ‚Äî previously `human.subjects` in the relay DB, sourced from `peers.json` ‚Äî is now the roster SoT `packages/operations/nats/fleet-roster.json` (¬ß2.6), read by the peer-agent at startup. Gate: executable tests for (a) publisher-not-owner rejected, (b) recipient-not-owner rejected, (c) `subject != null && in_reply_to != null` rejected, (d) subjected `presence_update`/`permission_*` rejected, (e) `subject != null && to == @team` rejected ‚Äî all ownership decisions read from `fleet-roster.json`, reproducing `routes/messages.ts` + `routes/stream.ts` chokepoint semantics app-side. The single shared matcher (`namespaceOf`/`ownsNamespace`/`matchesInterest`) stays the sole implementation. **JetStream terminal handling for app-side-rejected task messages:** the relay's "never insert an invalid subjected row" guarantee (SUBJECT_ROUTING_SPEC ¬ß4.1) cannot apply once NATS has accepted a publish into the durable WorkQueue. So a DIRECT `task_dispatch`/`task_result` that passes the NATS ACL and enters the WorkQueue but FAILS the app-side ACL (bad publisher/recipient namespace ownership) MUST be validated BEFORE processing and then `term`'d (JetStream `term` ‚Äî permanently removed from the WorkQueue, never redelivered) and written to an app-level audit log ‚Äî never `nak`'d (which would redeliver and starve valid tasks). (A `to.team` task never reaches the WorkQueue at all ‚Äî it is rejected at the sender/grammar layer per ¬ß2.6/AC13 and only direct tasks are ever published to the stream ‚Äî so it is out of scope for this terminal handling.) Gate: a test publishes an invalid DIRECT task (recipient-not-owner) into the stream and asserts it is `term`'d + audited and does NOT redeliver or block valid tasks behind it.
- **AC2b ‚Äî Subject-derived sender identity (anti-spoof):** peer-agent MUST derive trusted sender from authenticated publish subject (`fleet.<handle>.>`), never from envelope `from`; spoofed `from` in envelope is rejected or overridden.
- **AC3 ‚Äî `$SYS` account separation:** `$SYS` becomes an explicit admin account, not implicit/default anonymous.
- **AC4 ‚Äî Reconnect overflow is never silent:** publishes during disconnect/buffer overflow must surface explicit failure or be queued to durable app outbox semantics.
- **AC5 ‚Äî Permanent dedup:** repeat `task_dispatch` after the `Nats-Msg-Id` window (2 min) must still be deduped by KV.
- **AC6 ‚Äî Two-tier kind routing:** exactly six kinds are assigned to either core NATS or JetStream backfill tiers, with explicit matrix and tests.
**Tier matrix:**
| Kind | Delivery tier | Backfill policy |
|---|---|---|
| `chat` | core NATS pub/sub | no backfill |
| `presence_update` | core NATS pub/sub | no durable task replay needed |
| `permission_request` | core NATS request-reply / dedicated subject | immediate, no durable task queue |
| `permission_verdict` | core NATS request-reply / dedicated subject | immediate, must keep `in_reply_to` |
| `task_dispatch` | JetStream `WorkQueuePolicy` stream (per-handle DIRECT only; `@team` hard-rejected ‚Äî SUBJECT_ROUTING_SPEC R1, ¬ß2.6/AC13) | durable backfill for disconnected recipients |
| `task_result` | JetStream `WorkQueuePolicy` stream | durable backfill for disconnected result handlers |

- **AC7 ‚Äî Presence hybrid:** presence heartbeat is source-of-truth; `$SYS` events are only accelerants and may be dropped.
- **AC8 ‚Äî nats.js v3 client split packages only:** transport uses `@nats-io/transport-node`, `@nats-io/jetstream`, `@nats-io/kv`; no monolithic v2 `nats` package.
- **AC9 ‚Äî No KV TTL lease assumption:** no logic in the plan binds correctness to KV TTL lease semantics; arbitration substrate remains CAS+heartbeat+revision staleness and explicitly forbids correctness-on-TTL assumptions. Plan also treats KV residuals as by-design constraints: `R3` inconsistent reads, `Get/Watch` missed keys, and default `history=1` watcher behavior.
- **AC10 ‚Äî No leafnode federation and no `Nats-Request-Info` trust:** configuration ships with no `leafnodes` block and no protocol flow that trusts the header as identity.

## 2.6 NATS subject & ACL scheme (reproduces from-stamp + subscribe chokepoint)

The survey's illustrative `publish: ["fleet.<handle>.>"]` / `subscribe: ["fleet.>", "_INBOX.>"]` is INSUFFICIENT: a broad `subscribe: fleet.>` lets any peer read every other peer's traffic, breaking the relay's subscribe chokepoint ("only the addressed recipient receives" ‚Äî architecture.md ¬ß5.5, SUBJECT_ROUTING_SPEC ¬ß4.3). This plan pins a **recipient-addressed subject scheme**:

- **Wire subject:** every envelope publishes to `fleet.<sender>.to.<recipient>.<kind>` ‚Äî for a `@team` broadcast the recipient token is `team`. The `<sender>` token is the SECOND token.
- **Publish ACL (anti-spoof, AC2/AC2b):** each user's ENVELOPE publish is restricted to `fleet.<handle>.>` ‚Äî a peer may publish envelopes ONLY under its own sender handle, so the SECOND subject token is a NATS-enforced true sender identity (the ACL itself IS the from-stamp; the peer-agent never needs to independently verify the principal). The receiving peer-agent derives `from` from that token, NEVER from the envelope `from` field. **JetStream/KV control subjects are granted separately and minimally:** the same user also gets the scoped `$JS.API.>` / `$JS.ACK.>` and KV-bucket subjects required to publish to / ack the WorkQueue stream and do KV CAS (AC5/AC6/AC9), plus its own `_INBOX.<nuid>.>` for JS API replies. The "EXACTLY `fleet.<handle>.>`" constraint (AC2) applies to the ENVELOPE lane; the JS/KV control grants are a distinct, separately-tested minimal allow-list (not `fleet.>`, not wildcard `>`).
- **Subscribe ACL (subscribe chokepoint, AC12):** the ENVELOPE subscribe lane is exactly `["fleet.*.to.<handle>.>", "fleet.*.to.team.>"]` ‚Äî a peer receives ONLY envelope messages addressed to its own handle or to the team lane; it CANNOT subscribe to another peer's `to.<other>` traffic. The user ALSO gets a minimal, separately-tested CONTROL subscribe grant for its OWN `_INBOX.<nuid>.>` (to receive JetStream/KV API replies) ‚Äî this is NOT a broadening of the envelope lane and MUST NOT be a bare `_INBOX.>`. Deny-by-default otherwise. **Self-delivery drop:** because a broadcaster also matches its own `fleet.*.to.team.>`, the peer-agent applies an app-side receive gate that DROPS any message where `recipient == team && sender == localHandle` ‚Äî reproducing the relay's "fan out to every online handle EXCEPT `from`" (architecture ¬ß5.5).
- **Broadcast (`to.team`) lane is core-NATS `chat`/`presence_update` ONLY.** The peer-agent HARD-REJECTS any `to.team` message whose kind is NOT `chat`/`presence_update` ‚Äî i.e. `task_dispatch`, `task_result`, `permission_request`, `permission_verdict` to `@team` are all rejected (AC13). In particular `task_dispatch` is ALWAYS a per-handle direct DM, NEVER `@team` (SUBJECT_ROUTING_SPEC R1 ‚Äî a subjected/`task_dispatch` `@team` was a hard 400 on the relay); a hub dispatching to a namespace fans out ONE direct `fleet.<sender>.to.<owner>.task_dispatch` per roster owner. This keeps the JetStream `WorkQueuePolicy` stream (single-consumer, not broadcast) carrying only direct task messages, never a `to.team` task.
- **Reserved broadcast token:** the recipient token `team` is RESERVED for the broadcast lane; no fleet handle may be `team` (else it would collide with `fleet.*.to.team.>`). Config/roster tests reject a handle equal to `team` (and the shared `HANDLE_REGEX`/roster validation is extended to forbid it). `@team` in the envelope `to` maps to the wire recipient token `team`.
- **Wire-subject grammar gate (`parseFleetSubject`, AC13):** every inbound wire subject MUST parse as exactly the 5 tokens `fleet.<sender>.to.<recipient>.<kind>` with `<kind>` one of the six envelope kinds. The `<sender>`-token = authenticated-principal guarantee is enforced by the NATS **publish ACL** (`publish.allow = fleet.<handle>.>` means a peer physically cannot publish under another handle) ‚Äî it is an integration-test assertion against NATS, NOT a peer-agent check (the peer-agent cannot observe the publish principal and must NOT rely on `Nats-Request-Info`, AC10). The peer-agent's job is to: parse the grammar (reject malformed / extra-token subjects); DERIVE `from` from the `<sender>` token; and reject if `<recipient>` ‚âÝ the local handle (or `team`), or `<kind>` ‚âÝ the JSON `Envelope.kind`, or `<recipient> == team && <kind> ‚àâ {chat, presence_update}`. This makes the wire subject ‚Äî not the forgeable envelope body ‚Äî the identity/routing authority.
- **Request-reply (permission lanes) rides the SAME scheme, NOT raw `_INBOX`:** a `permission_verdict` is published to `fleet.<responder>.to.<requester>.permission_verdict` (inside the responder's own publish prefix), so publish-allow stays EXACTLY `fleet.<handle>.>` ‚Äî no `_INBOX.>` grant is needed and no responder gains publish authority outside its prefix. Reply correlation uses the envelope `in_reply_to` (unchanged), not a NATS inbox token.
- **Roster + namespace-ownership SoT (replaces `peers.json`/`human.subjects`):** the app-side envelope-subject ACL (AC11) needs an authority for who owns which namespace, which the retired relay read from `human.subjects`. This plan ships that authority as a new **`packages/operations/nats/fleet-roster.json`** (per-handle `owned`/`interest` namespaces + display name), distributed to every host and consumed by the peer-agent for `list_peers`, the inbound sender gate, and AC11's publisher/recipient ownership checks. It is the single membership + ownership SoT after `peers-file.ts` retires; a relay restart's re-seed discipline is replaced by re-reading the roster on peer-agent (re)start. **Roster‚áîNATS-users equality (AC12 gate), FLEET users only:** the roster handles and the `nats-server.conf` FLEET NKey users MUST be the exact same set ‚Äî `nats-config.test.ts` fails on any orphan fleet NKey user (in conf but not roster) or roster-only handle (in roster but no NKey user). The privileged `$SYS`/admin credential (AC3) is NOT a fleet user: it is explicitly EXCLUDED from `fleet-roster.json`, from the equality check, and from `list_peers`. `list_peers` returns the roster (fleet) handles annotated with heartbeat state, NOT a presence-discovered membership.

## 4. Phases

### Phase 0 ‚Äî NATS control-plane foundation in parallel with relay (size: H)
Goal: ship a hardened NATS control plane while keeping relay running for rollback.

Done when: `packages/operations/systemd/hangar-bridge-relay.service` and `packages/operations/systemd/install-relay.sh` can run a NATS unit side-by-side against the new `packages/operations/nats/nats-server.conf`, the roster SoT `packages/operations/nats/fleet-roster.json` (¬ß2.6) is shipped, and AC1, AC2, AC3, AC10, AC12 gates are proven in CI (`nats-config.test.ts`); relay remains untouched for production traffic during this phase.

Resolved operational decisions (were open questions): **NKey secret storage** ‚Äî each host's NKey seed lives at `~/.config/hangar-bridge/nats/<handle>.nk` mode `0600` (mirrors the existing `~/.config/hangar-bridge/secret` storage in architecture ¬ß4); the server config references public NKeys only (private seeds never leave the host, per the NKeys challenge-response model). **Rotation** ‚Äî rotate by generating a new seed, updating the user's public `nkey` in `nats-server.conf`, and `systemctl reload` (a reload drops live connections cleanly, mirroring the relay re-seed discipline). A CI test asserts the config references no private seed material and no plaintext credential.

Rollback trigger and step: if any hardening gate fails, stop NATS rollout and continue relay-only operation by uninstalling/disable the new unit. No source-path changes are introduced in this phase.

### Phase 1 ‚Äî Peer-agent transport seam and anti-spoof hardening (size: H)

## 6. Risks + inversion

- **What guarantees failure of AC2b:** if peer-agent trusts envelope `from`, a compromised peer can forge sender identity within its ACL namespace. The inversion gate is the explicit reject/rewrite test in AC2b.
- **What guarantees failure of AC6:** if kind matrix omits a kind or misroutes one lane, delivery invariants or backfill semantics drift; inversion check is matrix exhaustiveness and replay-path tests.
- **What guarantees failure of AC5:** if KV is treated as TTL cache, duplicate tasks can re-dispatch after timeout; inversion check is replay test beyond `Nats-Msg-Id` window.
- **What guarantees failure of AC7:** if `$SYS` events are treated as ground truth, a missed event causes permanent presence poisoning; inversion check is `$SYS`-dropped/stale-heartbeat test.
- **What guarantees failure of AC10:** if leafnode is configured or `Nats-Request-Info` is trusted, federation threat model becomes implicit. Inversion check is config scan and runtime header-trust review.

## 7. Out of scope

- Core arbitration protocol (capability negotiation, priority/interruption negotiation, offer/counter-offer rounds, binding allocation semantics) is out-of-scope as a protocol to be authored later. This plan only creates the substrate: KV/CAS, request-reply transport, and namespace-safe transport path.
- Leafnode federation is deferred and blocked by AC10 until a separate threat-model and identity plan is approved.
- nsc/JWT full auth machinery is out-of-scope; static NKeys are accepted for fixed 2‚Äì5 host fleet.
- Multi-node JetStream clustering is out-of-scope; AC1 fixes single-node R1 for this phase.
- **Audit-replay / message-history tooling** on top of JetStream is out-of-scope; the task stream is strict `WorkQueuePolicy` (Phase 2). A future durable-history stream is a separate plan.
- **Durable / cross-restart correlation memory** is out-of-scope. Task correlation stays peer-agent-local TTL state (architecture ¬ß5.6, unchanged); a peer-agent restart loses in-flight correlation exactly as today. The KV substrate (Phase 3) is dedup-only, not a correlation store.
